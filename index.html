fabric搭建一个牛奶溯源系统

发表于 2020-06-05 | 分类于 hyperledger > fabric | | 阅读次数:
字数统计: 1k | 阅读时长 ≈ 3

食品溯源是区块链的一个不错的应用点，通过将食品的整个生产流程记录在链上，能够很好的治理食品安全问题。
这次更新一下fabric网络手动搭建的的第二部分，链码的安装调用
本文依旧主要参考《区块链开发实战》第12章内容
该博客所引用代码地址

系统架构

整过牛奶的生产过程被简化为三个步骤，奶牛场养殖，加工厂加工，销售终端出售。

所以我们的区块链网络架构就是由三个组织构成的。
分别是dairyfarm，machine和salesterminal。
另外每个组织要上链自己的信息，销售终端还应该可以查看牛奶的溯源信息。
所以应该有三个chaincode(智能合约)来实现各种功能。
另外还需要一个终端来输入各种调用命令。因为仅仅是输入fabric的命令，非常的繁琐。

网络搭建

前面已经介绍了整个系统的基本构成。为了模拟出这个环境，我们需要搭建一个有三个组织的区块链网络。并且每个组织分配至少一个节点。
如何搭建网络环境这里不做解释，请参考我的另一篇博客

链码的编写和安装

需要编写三个链码，origin_dairyfarm用来上链奶牛的一些信息；origin_machining用来上链工厂的加工分配信息；origin_salesterminal用来上链销售信息和查询溯源信息。
最终的目的就是对于特定的一瓶牛奶，能够查询出其从奶牛到奶瓶的整个过程。
书上的链码是用go语言写的，没有找到电子版，我码了一份，其中有一些名称的修改。代码地址请看开头链接。
链码编写好后需要打包并安装在三个组织peer节点，因为他们都是背书节点。
关于如何打包和安装请参考另一篇博客

nodejs客户端的编写

为了方便开发者编写fabric的应用，他们提供响应架构的sdk。本文采用nodejs搭建客户端。

下载fabric-sdk-node

去github上下载fabric-sdk-node到本地
在解压后的文件中执行

npm install

安装依赖。

nodejs客户端调用链吗的主要逻辑就是

1.启动服务等待命令
2.接收命令
3.加载帐号访问fabric接口
4.发送调用提案请求
5.发送调用求情
6.返回调用结果

其中第3部需要用的cryptogen生成的帐号。此外还可以通过部署fabric-ca-server服务器来生成帐号。

将编写好的fabricservice.js(接口的封装)和mainorigin.js(主逻辑)放入fabric-sdk-node文件夹中，然后执行:

http://localhost:3000/init

开启服务。

奶牛场上链操作举例:

http://localhost:3000/diaryfarm?parms=food //奶牛喂食

加工场上链操作举例:

http://localhost:3000/machining?parms=canned //机器灌装

销售终端上链操作举例：

http://localhost:3000/salesterminal?parms=distribution_time //机器灌装

溯源查询举例：

http://localhost:3000/getmilkhistory

执行效果截图：

可以看到cow_002由machining_002生产出来，生产改奶得奶牛有很多喂食操作。

问题与总结

在执行链码时，如果命令参数给的不够，会导致链码镜像崩溃关闭，查看日志原因是越界导致。所以应在编写链码时所处一定的判断，并给出一定的返回提示，防止链码镜像崩溃。
另外链码逻辑和客户端的编写为了做出简单的效果演示，逻辑十分不严密，很多地方是写死的，比如客户端默认就是查询cow_001（我改了下）的溯源信息，且上链时信息必须匹配，不然无法溯源。总之就是链码逻辑只考虑简单的演示，并没有任何容错可言。

这次更新一下fabric网络手动搭建的的第二部分，链码的安装调用
本文依旧主要参考官方文档

链码的安装

链码依赖下载

链码的编写支持，go,js,java多种语言。本文以go为例。官方提供了一个简单的记录车子信息的链码demo。在fabric的chaincode文件夹下可以找到。

首先只有go写的链码是不够的。链码所调用的api所在的包文件需要进行拉取。
过程也很简单。
找到go目录下的fabric.go文件，打开可以看到其依赖的包，这两个是最常用的fabric包。

"github.com/hyperledger/fabric-chaincode-go/shim"
"github.com/hyperledger/fabric-protos-go/peer"

首先在链码fabcar.go同级目录下使用命令：

go init

这样会在该文件夹下得到一个go.mod文件
然后执行：

sudo GO111MODULE=on go mod vendor

这样fabcar.go所依赖的包就会被下载到vendor文件夹下。

链码打包

在拉取好依赖后，需要将链码打包才能安装在peer节点上。
打包命令是：

peer lifecycle chaincode package fabcar.tar.gz --path ../chaincode/fabcar/go/ --lang golang --label fabcar_1

打包的过程会进行一些语法性检查。
这里面的几个参数。--lang golang代表使用的语言是go，--path代表go文件所在的文件夹。

链码的安装

链码需要根据背书策略安装在所有需要安装的节点上。我们这里是peer.org1和peer0.org2。首先需要预置环境变量，例如：

export CORE_PEER_TLS_ENABLED=true
export CORE_PEER_LOCALMSPID="Org1MSP"
export CORE_PEER_TLS_ROOTCERT_FILE=${PWD}/organizations/peerOrganizations/org1.example.com/peers/peer0.org1.example.com/tls/ca.crt
export CORE_PEER_MSPCONFIGPATH=${PWD}/organizations/peerOrganizations/org1.example.com/users/Admin@org1.example.com/msp
export CORE_PEER_ADDRESS=localhost:7051

其中CORE_PEER_TLS_ROOTCERT_FILE如果没有开启tsl则不需要预置。
然后又执行安装命令：

peer lifecycle chaincode install fabcar.tar.gz

链码批准和提交

链码在被安装后，需要先被所有背书借点批准后，再由一个节点提交，才能被正常调用。

同样需要的预置环境变量，和上面的相同，不再重复。
查看已安装的链码：

peer lifecycle chaincode queryinstalled

返回内容：

Installed chaincodes on peer:
Package ID: fabcar_1:69de748301770f6ef64b42aa6bb6cb291df20aa39542c3ef94008615704007f3, Label: fabcar_1

将包id存成预置环境变量：

export CC_PACKAGE_ID=fabcar_1:69de748301770f6ef64b42aa6bb6cb291df20aa39542c3ef94008615704007f3

执行批准命令：

peer lifecycle chaincode approveformyorg -o localhost:7050 --ordererTLSHostnameOverride orderer.example.com --channelID mychannel --name fabcar --version 1.0 --package-id $CC_PACKAGE_ID --sequence 1 --tls true --cafile ${PWD}/organizations/ordererOrganizations/example.com/orderers/orderer.example.com/msp/tlscacerts/tlsca.example.com-cert.pem

其中，如果未开启tsl，则不需要--ordererTLSHostnameOverride和--tls两个选项。--package-id选项为刚才预置的环境变量。name就是链码的名称。
每个背书节点都需要对链码进行批准。
可以通过下面的命令，查看背书借点批准状态:

peer lifecycle chaincode checkcommitreadiness --channelID mychannel --name fabcar --version 1.0 --sequence 1 --tls true --cafile ${PWD}/organizations/ordererOrganizations/example.com/orderers/orderer.example.com/msp/tlscacerts/tlsca.example.com-cert.pem --output json

在所有背书节点都批准后，由一个节点提交该链码，命令：

peer lifecycle chaincode commit -o localhost:7050 --ordererTLSHostnameOverride orderer.example.com --channelID mychannel --name fabcar --version 1.0 --sequence 1 --tls true --cafile ${PWD}/organizations/ordererOrganizations/example.com/orderers/orderer.example.com/msp/tlscacerts/tlsca.example.com-cert.pem --peerAddresses localhost:7051 --tlsRootCertFiles ${PWD}/organizations/peerOrganizations/org1.example.com/peers/peer0.org1.example.com/tls/ca.crt --peerAddresses localhost:9051 --tlsRootCertFiles ${PWD}/organizations/peerOrganizations/org2.example.com/peers/peer0.org2.example.com/tls/ca.crt

提交之后，可以查看提交状态：

peer lifecycle chaincode querycommitted --channelID mychannel --name fabcar --cafile ${PWD}/organizations/ordererOrganizations/example.com/orderers/orderer.example.com/msp/tlscacerts/tlsca.example.com-cert.pem

如果如下，则代表提交成功：

Committed chaincode definition for chaincode 'fabcar' on channel 'mychannel':
Version: 1, Sequence: 1, Endorsement Plugin: escc, Validation Plugin: vscc, Approvals: [Org1MSP: true, Org2MSP: true]

链码升级

当对链码进行bug修复或者更新时需要进行链码升级，其实和安装过程基本一致

首先打包好新的安装包
然后安装，批准，提交
需要注意更新预置环境变量–package-id的值
同时批准和提交时的选项--version变成2.0，--sequence变成2.其他的不变。
执行新的链码命令来添加一个条目：

peer chaincode invoke -o localhost:7050 --ordererTLSHostnameOverride orderer.example.com --tls true --cafile ${PWD}/organizations/ordererOrganizations/example.com/orderers/orderer.example.com/msp/tlscacerts/tlsca.example.com-cert.pem -C mychannel -n fabcar --peerAddresses localhost:7051 --tlsRootCertFiles ${PWD}/organizations/peerOrganizations/org1.example.com/peers/peer0.org1.example.com/tls/ca.crt --peerAddresses localhost:9051 --tlsRootCertFiles ${PWD}/organizations/peerOrganizations/org2.example.com/peers/peer0.org2.example.com/tls/ca.crt -c '{"function":"createCar","Args":["CAR11","Honda","Accord","Black","Tom"]}'

peer chaincode query -C mychannel -n fabcar -c '{"Args":["queryAllCars"]}'

可以成功查询到刚刚添加的条目。

3.简单网络的搭建

3.1主要流程

搭建一个fabric的建议网络的主要流程是：

1.创建证书文件
2.创建创世区块、通道区块、锚点文件
3.启动orderer节点和peer节点
4.创建通道和加入节点加入

另外列出创建网络至关重要的几个配置文件：
cryto-config.yaml，configtx.yaml，core.yaml，orderer.yaml
这几个文件的配置要格外小心，错误往往出自这里。

3.2.创建证书文件

在进行一切之前首先建议将fabric-sample/bin/目录加入到path环境变量，因为后面要每一步使用的命令的可执行文件都在这个文件夹中或者，每当开始一个新的终端时，执行:

export PATH= \$PATH:\$fabric-sample\bin

这里创建的是fabric网络的证书文件。主要依据配置文件cryto-config.yaml
这个文件的内容末班可以通过执行：

cryptogen showtemplate

得到。这里给出样例。

  - Name: Orderer
    Domain: mynetwork.com
    EnableNodeOUs: false

    Specs:
      - Hostname: orderer


PeerOrgs:

  - Name: Org1
    Domain: org1.mynetwork.com
    EnableNodeOUs: false

    Template:
      Count: 2
    Users:
      Count: 3

  - Name: Org2
    Domain: org2.mynetwork.com
    EnableNodeOUs: false
    Template:
      Count: 2
    Users:
      Count: 2

这个文件的内容较少，也不复杂。
为了便于介绍，我们创建一个自网络的文件夹名字叫mynetwork，在这个文件夹下创建cryto-config.yaml并写入以上内容。
之后进入mynetwork文件夹执行命令

cryptogen generate --config=crypto-config.yaml --output ./crypto-config

然后在这个目录下就生成了crypto-config文件夹，这个里面就是我们需要的各种证书文件。

1.添加 docker group

$ sudo groupadd docker

2.将用户加入该 group 内

sudo apt-get update

$ sudo usermod -aG docker $USER
# 或者使用下面命令
$ sudo gpasswd -a ${USER} docker

3.重启服务

$ sudo service docker restart
# 或者
$ sudo /etc/init.d/docker restart

$ newgrp - docker

CharacterController

首先看一下CharacterController的官方定义：
A CharacterController allows you to easily do movement constrained by collisions without having to deal with a rigidbody.
A CharacterController is not affected by forces and will only move when you call the Move funtion. It will then carry out the movement but be constrained by collisions.

大概的意思就是角色控制器可以使我们控制物体的移动而不必使用刚体，只有在调用move函数时，物体才会移动，而不受任何其他力的影响(包括重力)

位移控制

定义一些预设变量

public Transform m_transform;  
CharacterController m_ch;      //角色控制器
public float m_movSpeed=5.0f;  //移动速度
float m_gravity = 9.8f;        //重力
public float m_jumpSpeed = 10.0f;    //跳跃初始速度
float move_y = 0;              //y方向位移值(配合重力和跳跃
Transform m_camTransform;      //摄像机

public float m_camHeigth = 3.0f;   //摄像机高度偏移

监听WASD用户输入实现角色移动控制

其中your_email@youremail.com是你的邮箱

ssh -T git@github.com

Hi xxx! You've successfully authenticated, but GitHub does not # provide shell access.
